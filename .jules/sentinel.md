## 2024-04-12 - DOM-based XSS via innerHTML and href Injection
**Vulnerability:** The application was directly constructing HTML strings using unsanitized external data (from JSON fetch and CSV parsing) and inserting them into the DOM via \`innerHTML\`. In addition, URLs derived from external data were being directly assigned to \`href\` attributes without protocol validation, allowing for potential \`javascript:\`, \`data:\`, or \`vbscript:\` execution.
**Learning:** Entity bypass can occur when untrusted data is injected directly into \`innerHTML\`. Sanitization helpers must explicitly escape HTML entities (\`&\`, \`<\`, \`>\`, \`"\`, \`'\`) and string-cast to prevent array bypass attacks. For URLs, simple string checks (like \`.startsWith\`) are insufficient due to whitespace bypasses; the \`URL\` API should be used to parse the URL and robustly check the \`.protocol\` property.
**Prevention:** Always sanitize inputs before injecting them into the DOM via \`innerHTML\` by using an \`escapeHTML\` helper. For dynamically constructed URLs, always parse and validate the URL using the \`URL\` object and restrict protocols to safe ones (like \`http:\` and \`https:\`), replacing invalid URLs with a safe fallback (e.g., \`#\`).
